# Release signing (auto-update)

> Source of truth for **how JustDownload signs its GitHub releases** so the opt-in update checker
> (`JustDownload.Core/Updates`, TASK-080) can verify them before downloading or launching anything.

## Policy

Every published release's `checksums.txt` (a sha256sum-style manifest of the release assets) is signed
with an **ECDSA P-256** keypair using only the .NET base class library (`System.Security.Cryptography`) —
no new dependency, no external signing service. The app embeds only the **public** half
(`JustDownload.Core/Updates/UpdateSigningKey.cs`); the **private** half is a GitHub Actions repository
secret used solely by `.github/workflows/release.yml`.

**The private key must never be generated by, seen by, or handed to an AI coding agent.** It must be
generated by a human maintainer, offline, and go straight into two places:

1. `.github/workflows/release.yml`'s signing step, via the `UPDATE_SIGNING_PRIVATE_KEY` repository secret.
2. Nowhere else. Not a file in this repo, not a chat log, not a commit.

Until that's done, `UpdateSigningKey.ProductionPublicKeyBase64` stays an empty placeholder and the update
checker **fails closed**: it reports "not configured" and never makes a network call, let alone trusts or
downloads anything (see `UpdateCheckerTests.CheckAsync_ReportsNotConfigured_AndMakesNoNetworkCall_WhenPublicKeyIsThePlaceholder`).

## Generating the production keypair (do this yourself, offline)

Run this **once**, on a machine you trust, disconnected from anything an agent can read. The snippet below
works as a scratch `dotnet-script` file, or paste it into a throwaway console app you delete afterward —
either way, nothing about this needs a new project added to `JustDownload.sln` (§4 of `CLAUDE.md`: no bloat
for a one-time, human-run script).

```csharp
using System.Security.Cryptography;

using ECDsa key = ECDsa.Create(ECCurve.NamedCurves.nistP256);

// The PUBLIC half: paste this into JustDownload.Core/Updates/UpdateSigningKey.cs
// (UpdateSigningKey.ProductionPublicKeyBase64) and commit it — it's not a secret.
Console.WriteLine("PUBLIC KEY (paste into UpdateSigningKey.ProductionPublicKeyBase64):");
Console.WriteLine(Convert.ToBase64String(key.ExportSubjectPublicKeyInfo()));

// The PRIVATE half: paste this into the UPDATE_SIGNING_PRIVATE_KEY GitHub Actions repository secret
// (Settings → Secrets and variables → Actions → New repository secret) and then delete it from
// wherever you ran this script. NEVER commit it, NEVER paste it into a chat with an AI agent.
Console.WriteLine();
Console.WriteLine("PRIVATE KEY (paste into the UPDATE_SIGNING_PRIVATE_KEY repo secret, then wipe this output):");
Console.WriteLine(Convert.ToBase64String(key.ExportPkcs8PrivateKey()));
```

Steps:

1. Save the snippet as e.g. `sign-key.csx` and run it with [`dotnet-script`](https://github.com/dotnet-script/dotnet-script)
   (`dotnet tool install -g dotnet-script && dotnet-script sign-key.csx`), or paste it into `Program.cs` of a
   throwaway `dotnet new console` app and `dotnet run`, then delete the scratch project.
2. Copy the **public key** line into `JustDownload.Core/Updates/UpdateSigningKey.cs`, replacing the empty
   `ProductionPublicKeyBase64` placeholder. Commit that change normally — the public key is not a secret.
3. Copy the **private key** line into a new GitHub Actions repository secret named
   `UPDATE_SIGNING_PRIVATE_KEY` (repo **Settings → Secrets and variables → Actions → New repository
   secret**). GitHub encrypts it at rest; it is never printed in logs.
4. Clear your terminal / delete the scratch script so the private key doesn't linger in shell history or a
   temp file.
5. If the key is ever compromised, generate a new pair and repeat steps 2–3 — old releases signed with the
   retired key simply stop being trusted by new app builds once the new public key ships.

## How a release gets signed (`.github/workflows/release.yml`)

Triggered by pushing a `v*` tag. It:

1. Builds the Windows installer via `build/build-installer.ps1` (TASK-159), producing
   `JustDownload-win-x64.msi` and `JustDownload-win-x64-Setup.exe` under `build/installer/out/`.
2. Computes each asset's SHA-256 and writes `checksums.txt` in sha256sum format
   (`<lowercase-hex-sha256>␠␠<filename>` per line, LF, no BOM — the exact bytes the signature covers).
3. Signs `checksums.txt`'s bytes with the `UPDATE_SIGNING_PRIVATE_KEY` secret
   (`ECDsa.SignData(bytes, HashAlgorithmName.SHA256)`, the IEEE P1363 signature format .NET uses by
   default) and writes the raw signature bytes to `checksums.txt.sig`.
4. Publishes a GitHub Release with the installer, `checksums.txt`, and `checksums.txt.sig` as assets.

The app's `UpdateChecker` mirrors step 3 in reverse: fetch `checksums.txt` + `checksums.txt.sig`,
`ECDsa.VerifyData` the signature against the embedded public key, and only then trust the hash `checksums.txt`
lists for the asset it's about to download — see `JustDownload.Core/Updates/UpdateChecker.cs` and
`JustDownload.Tests/Updates/UpdateCheckerTests.cs` for the exact contract (both a bad signature and a
correctly-signed-but-tampered manifest are rejected, independently of a hash mismatch on the downloaded
bytes).
